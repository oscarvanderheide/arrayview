## Attempt 1 — onAutoForward: "openPreview" (v0.0.8-ish)
Approach: Set `.vscode/settings.json` with `"remote.portsAttributes": { "8000": { "onAutoForward": "openPreview" } }`
Result:  FAILED — Simple Browser opens but with the devtunnel public URL
         (https://oscars-imachome-8000.euw.devtunnels.ms/...) which triggers
         GitHub OAuth login → 403 Forbidden. The devtunnel URL requires auth
         that Simple Browser can't handle.
Lesson:  onAutoForward "openPreview" opens the PUBLIC devtunnel URL, not localhost.
         This approach is fundamentally broken for private tunnels.

## Attempt 2 — onAutoForward: "silent" + IPC command execution
Approach: Set onAutoForward to "silent" (port forwards silently without opening).
         Then try to send `{"type": "command", "command": "simpleBrowser.show"}`
         over the VSCODE_IPC_HOOK_CLI Unix domain socket to trigger Simple Browser.
Result:  FAILED — IPC socket returned HTTP 404.
         The VS Code CLI IPC protocol does NOT support arbitrary command execution.
         It only supports a limited set of message types (openExternal, status, etc).
Lesson:  Cannot execute VS Code commands via IPC socket. The socket only supports
         "open" (openExternal) and a few other types. "command" type → 404.

## Attempt 3 — openExternal IPC with localhost URL
Approach: Send `{"type": "open", "uris": ["http://localhost:8000/..."]}` via IPC
         socket, hoping VS Code would open it in Simple Browser.
Result:  FAILED — VS Code converts localhost URLs to devtunnel URLs when using
         openExternal, then opens system browser with the devtunnel URL → OAuth error.
Lesson:  openExternal through IPC always resolves to the public tunnel URL.
         Cannot use IPC openExternal to open localhost in Simple Browser.

## Attempt 4 — VS Code Extension with signal file (v0.0.14)
Approach: Wrote a VS Code extension (arrayview-opener) that:
         - Activates on startup (onStartupFinished)
         - Watches ~/.arrayview/open-request.json via fs.watch + polling
         - When signal file appears, calls vscode.env.asExternalUri(localhost)
           to get the forwarded URL, then opens THAT URL in simpleBrowser.show()
Result:  PARTIAL — Extension activated successfully. Simple Browser opened.
         But it opened the devtunnel URL (from asExternalUri) → white screen / auth error.
Lesson:  asExternalUri() returns the public devtunnel URL. Passing that to
         simpleBrowser.show() has the same OAuth problem. Need to pass localhost
         URL directly to simpleBrowser.show() — it handles forwarding internally.

## Attempt 5 — asExternalUri for side-effect only, localhost to Simple Browser (v0.0.15)
Approach: Changed extension to:

         The port forwarding hadn't fully initialized when Simple Browser tried

## Attempt 15 — Root-cause analysis of persistent ERR_CONNECTION_REFUSED (2026-03-06)
Approach:
  - Reviewed the actual repo code instead of iterating on more delays.
  - Checked extension host placement and current opener logic.

Findings:
  - `vscode-extension/package.json` sets `"extensionKind": ["workspace"]`.
    In a tunnel session, that means the extension runs on the REMOTE machine
    (iMac), not in the local VS Code UI process on the MacBook.
  - The retry loop added in v0.3.0 calls `http.get()` against `localhost:PORT`
    from the REMOTE extension host. That only proves the ArrayView server is up
    on the iMac. It does NOT prove the client-side forwarded localhost proxy is
    ready on the MacBook.
  - Therefore the current "wait until /ping succeeds, then open Simple Browser"
    logic is checking the wrong thing. This explains why the retry loop can pass
    and Simple Browser can still fail with `ERR_CONNECTION_REFUSED`.
  - `Cmd+click` opening a devtunnel URL with 403 is unrelated noise: terminal link
    handling is converting the URL through the public tunnel path, which is not
    the same path used by `simpleBrowser.show(localhostUrl)`.

Conclusion:
  - `asExternalUri()` might still be useful as a side-effect to trigger port
    forwarding, but it is NOT sufficient evidence of a fix by itself, and more
    delay tuning in the current workspace-host extension is mostly guesswork.
  - The next real fix needs to target the CLIENT/UI side of the connection or use
    a VS Code API that explicitly guarantees the local forward is established,
    instead of probing remote localhost and assuming the client proxy is ready.
